# 방탈출 스토리 제작 시스템 (Story Production System) README

이 프로젝트는 수학 교육 방탈출 게임의 콘텐츠 품질을 극대화하기 위해 설계된 **데이터 및 프로토콜 기반의 스토리 제작 파이프라인**입니다.

단일 거대 프롬프트의 평균적인 이야기 출력 한계를 극복하기 위해, **관심사 분리(Separation of Concerns)** 원칙을 바탕으로 15개의 독립된 역할 및 명세 파일로 구성되어 있습니다.

---

## 1. 아키텍처 철학 (Philosophy)
- **설계와 집필의 분리**: `ARCHITECT`는 뼈대만 설계하고, 문장은 오직 `WRITER`만 작성합니다. AI가 개연성 없는 평면적 스토리를 쏟아내는 것을 원천 차단합니다.
- **비파괴적 국소 수정 (Review & Fix)**: 에러가 났을 때 전체 스토리를 갈아엎으면 설정이 완전히 깨집니다. 이 시스템은 문제 지점만 정밀 타격하여 Diff 형태로 수정합니다.
- **규격화된 통신 계약 (Data Contract)**: 각 에이전트 간 데이터 교환 시 YAML, JSON, Diff 포맷을 엄격하게 제한하여 시스템 안정성을 보장합니다.
- **게이트키퍼에 의한 승인**: `QA` 에이전트가 스토리 바이블 부합 여부와 수학 오류를 정밀 체크한 후에만 최종 배포됩니다.

## 2. 파일 구조 및 역할 요약

```text
story/
├── 01_SPEC.md            # [입력] 교과 단원, 퍼즐 수, 타겟 난이도 등 기획 명세
├── 02_GLOBAL_BIBLE.md     # [규칙] 시리즈 전체 공통 가치관, 금기사항 및 제약
├── 03_PROJECT_BIBLE.md    # [규칙] 이번 에피소드 고유 세계관, 캐릭터, 핵심 반전
├── 04_PIPELINE.md         # [설계] 파이프라인 프로세스 및 입출력 데이터 포맷 스펙
├── 05_ARCHITECT.md        # [프롬프트] 뼈대 설계 에이전트 (Output: YAML)
├── 06_WRITER.md           # [프롬프트] 대사 및 서사 집필 에이전트 (Output: Markdown)
├── 07_REVIEWER.md         # [프롬프트] 다중 관점 문제점 감지기 (Output: JSON)
├── 08_SCORE.md            # [프롬프트] 8대 객관적 지표 채점 루브릭 (Output: YAML)
├── 09_DIRECTOR.md         # [프롬프트] 수정 이슈 우선순위 분류 (Output: YAML)
├── 10_FIXER.md            # [프롬프트] 국소적 부분 수정 에이전트 (Output: Diff)
├── 11_QA.md               # [프롬프트] 체크리스트 품질 검수 및 통과 판정 (Output: YAML)
├── 12_EXAMPLES.md         # [학습] 퓨샷(Few-shot) 품질 대비 사례집
├── 13_STATE.md            # [추적] 전체 작업 Revision 및 수정 히스토리 로그
└── 14_METRICS.md          # [통계] 누적된 완성 시나리오의 통계 지표 데이터베이스
```

## 3. 핵심 워크플로우 (Workflow)

```text
[Step 1] 01_SPEC 및 03_PROJECT_BIBLE 작성
   │
[Step 2] 05_ARCHITECT 실행 ➔ 시나리오 설계도(YAML) 도출
   │
[Step 3] 06_WRITER 실행 ➔ 스토리 초고(Markdown) 도출
   │
[Step 4] 07_REVIEWER & 08_SCORE 동시 실행 ➔ 문제점 리스트(JSON) 및 점수(YAML) 확보
   │
[Step 5] 09_DIRECTOR 실행 ➔ 우선순위 분류 오더 시트 도출
   │
[Step 6] 10_FIXER 실행 ➔ 오더 시트를 반영한 부분 수정(Diff) 수행 & 13_STATE 업데이트
   │     (QA 통과할 때까지 Step 4~6 루프 진행)
   │
[Step 7] 11_QA 실행 ➔ 최종 통과(PASS) 여부 확인
   │
[Step 8] 완료된 스토리 배포 및 14_METRICS 데이터 누적 업데이트
```

## 4. 자주 묻는 질문 (FAQ)

**Q. AI가 자꾸 뼈대 설계를 시켰는데 대사까지 같이 작성해서 소설을 만들어버립니다.**
- A. `05_ARCHITECT.md`에 명시된 출력 계약(YAML만 출력) 규정을 다시 주입하고 프롬프트 지침 강도를 높이십시오. 문장 형태가 보이면 즉시 거절해야 합니다.

**Q. 수정(`FIXER`)을 돌렸더니 새로운 오류가 발생하거나 이전 설정이 깨집니다.**
- A. 수정 전 반드시 `13_STATE.md`와 `02_GLOBAL_BIBLE.md`를 함께 주입하여, 이전 히스토리와 고유 설정을 위반하지 않도록 콘텍스트 범위를 제한하십시오.
